The Use Case: Aviation Safety Analysis

Objective - Help safety analysts answer complex questions across multiple data sources with accurate, sourced, and trustworthy answers.

The Challenge

  • Safety analysts need answers from multiple sources
  • Questions range from simple facts to complex causal analysis
  • Compliance questions require cross-referencing regulations with incidents
  • Answers must be accurate, sourced, and trustworthy

Data Sources

📋

NTSB Incident Reports

Detailed accident investigations and findings

⚖

Federal Aviation Regulations and compliance requirements

📰

Aviation News

Current events and industry developments

The Approach: Agentic RAG

Core Idea - Instead of blindly retrieving documents, the system first understands what kind of question is being asked, then selects the right strategy to answer it.

🧠

Intent-Aware

Classifies questions as causal, compliance, factual, or comparative before retrieval

🎯

Multi-Source

Intelligently selects which data sources to query based on the question type

🧹

Context Refinement

Cleans and deduplicates retrieved content before generating answers

📑

Structured Output

Returns findings with sources, confidence levels, and explicit caveats

This is what makes it "agentic" — the system makes intelligent decisions at each step rather than following a fixed pipeline.

The Problem with Traditional RAG

Simple retrieval pipelines fall short for complex analytical questions

One-Size-Fits-All Retrieval

Every query uses the same retrieval strategy, regardless of whether you're asking about regulations, incidents, or comparisons.

No Query Understanding

The system doesn't know if you want causal analysis, compliance checking, or factual information.

Context Pollution

Retrieves redundant, irrelevant, or contradictory passages that confuse the generation model.

Unstructured Output

Returns free-form text without clear findings, sources, or confidence levels.

Result: Inconsistent answers, missed information, hallucinations, and inability to handle complex multi-part questions.

Traditional RAG vs Agentic RAG

Traditional RAG

  • Query → Embed → Retrieve → Generate
  • Single retrieval pass
  • All sources treated equally
  • Raw context concatenation
  • Unstructured text output
  • No quality control

Agentic RAG

  • Understand → Plan → Retrieve → Refine → Generate
  • Iterative retrieval with feedback
  • Intent-driven source selection
  • Context deduplication & synthesis
  • Structured, sourced findings
  • Sufficiency evaluation & caveats

Step 2: Context Refinement

Raw retrieval results are noisy — refinement fixes that

Deduplication Mode

Removes redundant passages while preserving original text

  • Identifies semantically similar passages
  • Keeps the most detailed/authoritative version
  • Preserves exact wording for citation
  • Best for: Compliance, Factual queries

Synthesis Mode

Compresses multiple passages into a coherent summary

  • Merges information from multiple sources
  • Eliminates contradictions
  • Creates unified narrative
  • Best for: Causal, Comparative queries

Why it matters: LLMs perform better with clean, focused context. Garbage in = garbage out.

Limitations of This Approach

The compression stage is a failure surface — stacking recall degradations before the reasoning model sees data

Lossy Compression

Dedup + synthesis transforms data, not just reduces it. The reasoning model operates on a distorted evidence set, not just a smaller one.

Semantic Dedup Danger

Near-duplicates often contain non-identical constraints. Removing one can erase edge conditions, qualifiers, or minority signals.

Synthesis ≠ Faithful Compression

Small models tend to normalize conflicting statements, drop minority signals, and smooth uncertainty into false clarity.

Error Amplification

Any hallucination or distortion introduced by the compression model becomes "ground truth" for the reasoning model.

Core Trade-off: We're trading token cost ↓ for information entropy ↑. If entropy rises too high, the reasoning model spends tokens recovering lost context, negating savings.

The Problem with Lossy Refinement

Concrete examples showing how information is lost

Dedup Mode Risks

Document A: "Engine failed at 3000 feet due to fuel contamination"

Document B: "Engine failed at 3000 feet, fuel system showed contamination signs from improper fuel storage"

Dedup keeps ONE, drops the other. But Document B has a critical detail about the cause of contamination that Document A lacks. That detail is now LOST.

Synthesize Mode Risks

• "Pilot reported engine roughness at 2500 RPM"

• "Mechanic found no evidence of engine issues"

• "Post-crash analysis showed carburetor icing"

Small model synthesis produces: "There were conflicting reports about engine condition"

Lost: The specific RPM, the mechanic's finding, the actual cause.

Filter Mode Solution

Same input → Score each chunk's relevance → Keep all above threshold with original wording.

The reasoning model sees ALL THREE chunks. It can: reconcile the conflict itself, note the mechanic found nothing (important context), connect carburetor icing to the reported roughness, and cite specific details in its answer.

Production Enhancements

Better pipeline shape: high recall retrieval → intelligent filtering → raw chunks to reasoning model

☑

Filter, Don't Rewrite

Use small model to select relevant chunks, not rewrite them. Keep original text intact for the reasoning model.

📈

Cross-Encoder Reranking

Score relevance with cross-encoders rather than summarize. Re-rank aggressively, keep top-K raw chunks.

📊

High-Recall First Pass

Retrieve K=large, then classify relevance, cluster loosely, drop clearly irrelevant. Pass minimally pruned chunks forward.

🧠

Let Reasoning Model Resolve

If two chunks partially overlap → keep both. Let the large model handle synthesis, reasoning, and conflict resolution.

Optimization Target: Minimize tokens removed per unit of information lost. Most "summarize before reasoning" pipelines violate this principle.
